## 1. API 配额风险

### 1.1 计划已识别
计划风险表第 1 行: "查询增加导致 API 配额超限 → 采集失败"。

### 1.2 风险量化分析

| 项目 | v6 现状 | v7 计划 | 变化 |
|------|--------|--------|------|
| chip_queries 查询数 | 3 条 | 6-8 条 | +100%~+167% |
| robot_queries 查询数 | 3 条 | 6-8 条 | +100%~+167% |
| 其他分组（不变） | 25 条 | 25 条 | 0% |
| **总查询数** | **31 条** | **37-41 条** | **+19%~+32%** |
| 每次查询 num_results | 5-10 | 5-10（未提及调整） | 0% |
| **每次运行总请求结果数** | ~180 条 | ~200-230 条 | +11%~+28% |

Exa API 的配额限制通常按以下维度计算:
- **月度查询次数** (search calls): +19%~+32%
- **月度结果返回条数** (results returned): +11%~+28%
- **并发限制**: 当前 max_concurrent_queries=5，查询数量增加后并发压力不变

### 1.3 缓解措施不足
计划中缓解措施仅 "监控 API 使用量，设置告警"，存在以下问题:

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| 未明确 Exa API 当前配额上限 | 无法判断 v7 是否超限 | 高 |
| 无配额预留/回退机制 | 超限后整个采集全量失败，没有降级方案 | 高 |
| 无请求节流/背压实现 | 代码中没有任何 rate limit 或 retry-backoff 逻辑 | 中 |
| 告警只是事后措施 | 不能防止采集失败 | 高 |

### 1.4 遗漏风险

**【遗漏 R1】同类查询间结果高度重叠导致隐性浪费**
芯片查询从 3 条扩到 6-8 条后，相邻查询（如 "AI chip GPU NVIDIA semiconductor compute latest" 和 "NVIDIA AMD Intel AI accelerator chip new release"）的返回结果高度重叠。Exa 按查询计费，重叠内容等于浪费配额。计划未考虑查询去重或查询合并策略。

**【遗漏 R2】新增数据源查询可能触发 Exa 站点级限制**
计划新增了 TechCrunch、IEEE Spectrum 等专项数据源查询（通过 site: 语法）。Exa 对单域名的爬取有内部速率限制，短期大量请求同一域名可能导致被限或返回空结果。代码中无应对此情况的处理。

**【遗漏 R3】无配额耗尽时的断路与降级策略**
如果月度配额在月底耗尽，系统应能自动降级（减少查询数、跳过非核心查询、使用缓存等）。当前实现和计划均无此设计。建议新增:
- 自动检测配额剩余（通过 API 响应头中的配额信息）
- 降级模式: 恢复 v6 查询数作为 fallback
- 缓存兜底: 配额耗尽时使用前一天缓存

### 1.5 改进建议
1. 在 config.yaml 中增加 `exa_quota_limit: { monthly_search_calls: X, monthly_results: Y }` 配置
2. 在 exa_collector.py 中增加请求节流（token bucket 或 sliding window）
3. 增加查询去重: 将 high-level 查询合并，减少重复结果
4. 增加配额耗尽降级逻辑: 优先保障核心查询，裁剪非核心查询

## 2. 聚类算法误判风险

### 2.1 计划已识别
计划风险表第 2 行: "聚类算法优化引入误判 → 聚类不准确"。

### 2.2 缓解措施不足

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| "充分测试"太笼统 | 无具体测试指标（精确率/召回率/边界用例） | 高 |
| "保留回滚机制"但未说明 | 回滚是指 git revert 还是配置开关？无实现细节 | 中 |
| 阈值变更缺乏定量基准 | 从固定 0.35 改动态阈值，但无验证基准数据 | 高 |

### 2.3 动态阈值风险分析

计划将 cross_source_threshold 从固定的 0.35 改为"动态阈值"。当前的 `cross_source_cluster()` 使用双重匹配:
1. Jaccard > threshold（配置值）
2. OR 共享有意义关键词 >= min_shared（配置值）

动态阈值的实现未在计划中描述，存在以下风险:

- **动态阈值的不当设计**可能导致:
  - 阈值过低 → 大量不相关文章被聚类在一起（误报）
  - 阈值过高 → 相关文章无法聚类（漏报）
  - 缺乏上下界约束 → 极端情况下无任何聚类或所有文章聚为一类
- **缺乏回归测试数据**: 没有标注好的测试集来验证聚类质量的变化

### 2.4 遗漏风险

**【遗漏 R4】增量聚类的一致性问题**
当前 `cross_source_cluster()` 每次对所有 items 重新聚类（"batch" 模式）。v7 新增因果链分析后，跨天的聚类一致性可能受到影响。例如:
- Day 1: 文章 A, B 聚为一类
- Day 2: 新增文章 C, D，但由于新关键词导致全局聚类漂移
- 结果: 同一话题在不同天的聚类结果不同，影响趋势追踪

**【遗漏 R5】跨域保留逻辑与聚类冲突**
计划中"不同域名的相似文章保留"逻辑与聚类算法的"相似度合并"逻辑存在内在矛盾:
- 去重器（dedup.py）将 Jaccard > 0.45 的文章视为重复并丢弃
- 跨源聚类器将 Jaccard > 0.35 的文章视为同类并聚合
- 当同域文章被去重后，跨源聚类器可能找不到足够的跨源信号
- 跨域保留逻辑如果强行保留已被去重丢弃的文章，会引入不一致

### 2.5 改进建议
1. 在实施计划中明确定义动态阈值算法（如 median + std 自适应）
2. 构建标注的小型测试集（~100 条，含清楚的正负例边界用例）
3. 明确聚类回滚机制（配置开关: `cross_source: { dynamic_threshold: true/false }`）
4. 解决去重器与聚类器阈值冲突的问题（统一阈值或明确两者关系）

## 3. 性能下降风险

### 3.1 计划已识别
计划风险表第 3 行: "因果链分析复杂度过高 → 性能下降"。

### 3.2 缓解措施不足

计划缓解措施 "限制分析深度，异步处理" 存在以下问题:

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| "限制分析深度"未量化 | 无具体深度值或复杂度上界 | 中 |
| "异步处理"与当前架构冲突 | main.py 是同步流水线，异步引入会增加复杂度 | 高 |
| 不考虑查询量增加带来的性能影响 | 19%-32% 的查询增加预期会线性增加采集耗时 | 中 |

### 3.3 当前性能基线推测

基于代码分析:

| 阶段 | v6 耗时估计 | v7 耗时预期 | 变化 |
|------|-----------|-----------|------|
| Exa 采集（31 查询 → 37-41） | ~31 × 2-5s = 62-155s | ~37-41 × 2-5s = 74-205s | +12-50s |
| 去重+聚类（~150 items） | < 1s | < 1s | 基本不变 |
| 因果链分析（新增） | 0 | 待定 — 完全取决于实现 | 新增 |
| 趋势追踪 | < 1s | < 1s | 不变 |
| **总计** | **~62-155s** | **~74-205s + 因果链耗时** | **+19%~+32%** |

query_timeout 当前 60s，但 ThreadPoolExecutor 包裹的是整个采集过程（含所有 31 条查询），超时阈值不够精细。

### 3.4 遗漏风险

**【遗漏 R6】因果链分析复杂度失控**
因果链分析（A→B 逻辑关系）如果采用全对全比较（所有 items 两两比较其因果关联），复杂度为 O(n²)。假设聚类后有 150 条 items，两两比较为 ~11,250 对。如果每对需要调用 LLM 或复杂 NLP 推理，耗时会从 <1s 飙升到数分钟。计划没有为因果链分析给出复杂度保证。

**【遗漏 R7】缓存一致性失效**
v6 使用 1800s（30 分钟）缓存。v7 新增因果链分析和跨源趋势追踪（3 天窗口）后:
- 趋势追踪依赖历史数据，缓存数据如果只存当天的结果，趋势窗口内的历史数据从哪里来？
- 如果缓存命中，因果链分析可能看不到最新的跨天数据
- 如果每次都重新计算，缓存的作用大打折扣
- main.py 的缓存策略未在计划中重新审视

**【遗漏 R8】并发采集的超时阈值不合理**
当前 `max_workers=1` 的 ThreadPoolExecutor 用于包裹每个采集器的超时控制。但实际 Exa API 调用是同步的，内部没有并发。query_timeout (60s) 需要覆盖所有 31 条查询（v7 为 37-41 条），而单条 Exa 查询可能耗时 2-10s。如果某条查询挂住，超时设置不当可能导致整个采集器超时。建议将超时细化到单条查询级别。

### 3.5 改进建议
1. 量化因果链分析的复杂度上界（如: 最多分析 Top-K 个聚类的因果链，每个聚类最多 N 个 item）
2. 明确异步处理的实现方式（celery / background thread / 或保持同步但有 timeout）
3. 将 query_timeout 细化到单条查询级别
4. 评估 30 分钟缓存 TTL 在 v7 场景下是否合理
5. 在 config.yaml 中增加因果链分析相关的性能配置项

## 4. 向后兼容性风险

### 4.1 计划中虽然列出了兼容性矩阵但未将其列入风险表

实施计划第 115-127 行有向后兼容性矩阵，但风险表（第 130-134 行）中完全没有列出向后兼容性相关的风险。这是一个严重遗漏。

### 4.2 兼容性矩阵的不足

兼容性矩阵列出了必须保留的字段，但存在以下问题:

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| 未明确 items 内字段兼容性 | items 内每条的结构（title/url/source/published/snippet/query_group/category）是否有变更？ | 高 |
| cross_platform_signals 增强无规范 | "可增强"太模糊，下游消费者无法判断新增字段 | 中 |
| trend_momentum 格式可能改变 | 当前输出是 {continuing, new} 各 15 个关键词列表，v7 计划支持 3 天窗口，可能改为对象格式 | 高 |
| 无版本号机制 | JSON 输出无 schema version，下游无法自动检测变更 | 中 |

### 4.3 遗漏风险

**【遗漏 R9】items 内字段变更导致下游解析失败**
当前 items 的结构（见 exa_collector.py 第 168-176 行）包含 title/url/source/published/snippet/query_group/category。如果 v7 新增了因果链字段（如 `causal_links`、`related_items`），下游消费者（飞书推送、前端展示）如果未适配会忽略新字段，但这不影响正常运行。但如果移除或重命名任何现有字段，会造成下游崩溃。

**【遗漏 R10】聚类输出结构变更**
`cross_source_cluster()` 当前返回的列表包含 topic/sources/item_count/sample_titles/urls。v7 增加因果链分析后，可能新增 `causal_chains`、`consensus_analysis` 等字段。如果下游依赖严格 schema 验证会报错。

**【遗漏 R11】趋势追踪数据格式变更**
当前 main.py 的 trend_momentum 输出是:
```json
{
  "trend_momentum": {
    "continuing": ["keyword1", "keyword2", ...],
    "new": ["keyword3", ...]
  }
}
```
但如果 v7 改为输出 trend_momentum.py 中的 momentum_data 对象（含 score/heat/streak），格式完全不同。这是一个破坏性变更。

**【遗漏 R12】飞书推送模板依赖**
如果飞书推送模板（如果存在）硬编码了 JSON 输出的嵌套结构，任何字段位置或嵌套深度的变化都会导致推送失败。计划应在 slice 4 中明确验证飞书推送端的兼容性。

### 4.4 改进建议
1. 明确 JSON 输出 schema version（新增 `schema_version: "v7.0"` 字段）
2. 在向后兼容性矩阵中补充 items 内字段的兼容性要求
3. trend_momentum 格式保持向后兼容（保留 existing 字段，新增字段以 additive 方式添加）
4. 在测试中增加兼容性测试用例，用 v6 的实际输出 schema 验证 v7 输出
5. 将向后兼容性风险明确列入风险表

## 5. 风险优先级评估

| 风险 | 影响 | 概率 | 优先级 | 当前优先级 |
|------|------|------|--------|-----------|
